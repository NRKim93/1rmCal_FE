# 트레이닝 프로그램 화면 설계 메모

- 작성일: 2026-07-19
- 상태: 검토용 초안
- 관련 백엔드 문서: `../../backend/docs/training-program-design.md`

## 목적

트레이닝 메인 화면의 현재 구현과 향후 프로그램 데이터 연동 방향을 정리한다. 기본 화면 구조는 유지하면서, 상단 프로그램 영역만 카드 슬라이드로 발전시키는 것이 목표다.

## 현재 화면 구조

```text
현재 진행중인 프로그램
  [트레이닝 카드 슬라이드]
  [페이지 인디케이터]

자유 트레이닝
  [자유 트레이닝 시작]

신규 트레이닝 생성
  [트레이닝 생성]
```

프리 트레이닝과 프로그램을 탭으로 나누지 않는다. 자유 트레이닝은 독립적인 빠른 시작 영역으로 유지하고, 슬라이드는 여러 프로그램 또는 프로그램 회차를 탐색하는 용도로 사용한다.

## 현재 구현

관련 파일:

- `src/app/trainingMain/page.tsx`
- `src/app/trainingMain/(components)/ProgramCard/ProgramCard.tsx`
- `src/app/trainingMain/(components)/ExerciseTable/ExerciseTable.tsx`
- `src/services/trainingMain.service.ts`
- `src/lib/types/training.ts`

현재 `GET /api/v1/training/getLatestHistory` 응답의 최근 5개 운동 세션을 카드로 변환한다.

- 좌우 스와이프로 카드 이동
- 인디케이터 클릭으로 카드 이동
- 카드 제목은 임시로 운동 날짜를 사용
- 카드 시작 시 선택한 `training_history`를 `sessionStorage`에 저장
- `/trainingMain/free?mode=program`으로 이동해 선택 기록을 불러옴

## 현재 데이터의 한계

상단 제목은 `현재 진행중인 프로그램`이지만 실제 데이터는 프로그램이 아니라 완료된 운동 기록이다. 백엔드 조회에 운동 모드 조건이 없으므로 자유 운동 기록도 카드에 노출된다.

현재 응답에는 다음 값이 없다.

- 실제 프로그램 ID와 프로그램명
- 프로그램 상태
- 현재 주차와 일차
- 전체 진행률
- 다음 회차의 목표 운동 구성
- 프로그램 운동과 자유 운동 구분

따라서 현재 화면에서 프로그램명과 진행률을 임의로 생성하지 않는다.

## 목표 카드 구성

백엔드 프로그램 API가 준비되면 각 슬라이드는 하나의 활성 프로그램을 나타낸다.

```text
[프로그램 아이콘] 프로그램 A
현재 6주차 · 3일차

진행률 81% [================----]

운동 종목          세트      목표
벤치프레스          4        8회
바벨 로우           4       10회
...

[6주차 · 3일차 시작]
```

여러 활성 프로그램이 있으면 `프로그램 A -> 프로그램 B` 순서로 스와이프한다. 프로그램이 하나면 인디케이터를 숨긴다.

## 예상 API 계약

프런트엔드에서 필요한 최소 응답 예시는 다음과 같다.

```json
{
  "data": [
    {
      "userProgramSeq": 12,
      "programSeq": 3,
      "programName": "스트렝스 8주",
      "status": "ACTIVE",
      "progressPercent": 62,
      "currentWeek": 5,
      "currentDay": 2,
      "nextSession": {
        "programDaySeq": 42,
        "exercises": [
          {
            "trainingCategorySeq": 1,
            "name": "스쿼트",
            "sets": 4,
            "targetWeight": 100,
            "weightUnit": "kg",
            "targetReps": 5,
            "restSeconds": 180
          }
        ]
      }
    }
  ]
}
```

프런트엔드는 `progressPercent`, `currentWeek`, `currentDay`를 계산하지 않고 응답 값을 표시한다.

## 화면 상태

### 로딩

- 프로그램 영역 높이가 급격히 바뀌지 않도록 카드 형태의 스켈레톤 표시
- 자유 트레이닝 시작은 프로그램 조회와 독립적으로 사용할 수 있어야 함

### 활성 프로그램 없음

- 상단에 빈 프로그램 상태 표시
- `신규 트레이닝 생성`으로 연결
- 자유 트레이닝 시작은 그대로 노출

### 조회 실패

- 프로그램 영역에 재시도 액션 제공
- 자유 트레이닝 시작은 차단하지 않음

### 활성 프로그램 여러 개

- 한 화면에 카드 하나 표시
- 좌우 스와이프와 인디케이터 제공
- 사용자가 보고 있는 카드의 시작 버튼은 해당 프로그램 및 회차 식별자를 전달

## 시작 흐름

프로그램 운동 시작 시 과거 기록 전체를 `sessionStorage`로 복사하는 현재 방식은 임시 구현이다.

목표 흐름:

1. 사용자가 프로그램 카드를 선택한다.
2. `userProgramSeq`와 `programDaySeq`를 기준으로 다음 회차를 요청한다.
3. 프로그램 목표 세트를 운동 입력 화면의 초기값으로 구성한다.
4. 완료 시 프로그램 운동 전용 API로 저장한다.
5. 성공 응답 후 해당 프로그램의 진행률과 다음 회차를 갱신한다.

페이지 새로고침에도 복구할 수 있도록 장기적으로는 URL 식별자 또는 서버 조회를 사용하고, `sessionStorage`는 보조 수단으로만 사용한다.

## 구현 순서

1. 신규 트레이닝 생성 화면의 프로그램 데이터 구조 확정
2. 프로그램 목록 및 다음 회차 API 계약 확정
3. 현재 날짜 기반 `TrainingSlide`를 프로그램 응답 타입으로 교체
4. 카드에 프로그램명, 진행률, 주차와 일차 표시
5. 프로그램 시작 요청과 자유 운동 시작 요청 분리
6. 빈 상태, 오류 상태, 스켈레톤 추가
7. 모바일 스와이프와 긴 종목명 표시 검증

## 이번 작업 범위

문서만 추가했다. 현재 트레이닝 메인 화면과 API 연결 동작은 변경하지 않았다.
